Often, when a rather large problem needs to be solved repeatedly, it is efficient to have a process to tackle it. Imagine a factory production line for example. Instead of building a car from scratch, it is more efficient to instead start from its base and have it travel down a line, with parts being added. This is similar to design patterns in software engineering. In both scenarios, it is a set of standardized procedures put in place to handle repeated, predictable problems. One way that the car example differs from design patterns is that in software engineering, we are not using a specific template. There is not a chunk of code that you can reuse over and over. Instead it is the idea of conceptual blueprints that help developers to write said code in a way that is faster, easier to understand, and comprehensive. 

Design patterns are important because patterns are easy to follow. It reduces complexity and allows for a standardized method to be used. This can help as it gives a template to a proven solution and prevents developers from having to “reinvent the wheel” every time they come across a problem. This is similar to the idea that you do not have to reinvent how to assemble a car every time that one passes down the assembly line. 

Design patterns that I have used is the top down method. It is the idea that you take a big problem and break it down into smaller pieces. Or, you take a github issue and break it up into subissues. This makes it more comprehensible and can let developers better understand what needs to be done and approximate time and difficulty estimations.
