Ethics in software engineering pertains to software systems and code that go on to affect the lives of other people. Even if the code or system was not made or intended to go out to the public or outside of your private domain, is it important to always keep in mind engineering ethics in addition to a personal code of ethics. Engineering ethics is a branch of ethics that concentrates on the moral principles and standards of behaviors that engineers should follow in their professional work. According to University of Pittsburgh Engineering, engineering ethics, “prioritizes public safety, health, and welfare, requiring engineers to act with integrity, honesty, and competence to maintain public trust and prevent harm to people or the environment.” In software engineering, the impacts that can be seen are not as straightforward as branches of engineering such as mechanical, civil, or electrical. But, with advancements to technology and the constant integration of new software and computation, its importance is growing. 


It is important to have ethics in not only engineering, but in all aspects of life. Without ethics, you reduce the common good. This is what leads to a decline in democracy and equity. Poet Percy Bysshe Shelley states in A Defence of Poetry, “The rich get richer and the poor get poorer” when referring to economic inequality. This still holds true today, and would be amplified without ethics. From learning about ethics in software engineering, I have learned about what it means to be mindful about the code that you write, and what consequences that it can have.


Agile project management in general is known as an approach to project management. The idea behind it is to break down big problems into smaller, more manageable chunks and repeat this process in cycles or iterations until the problem has been solved. In ICS314, we learned about a style of agile project management called issue driven project management. This style encourages identifying issues and solving them in real time. For example, bugs, errors, scope changes, etc. This is in comparison to a more traditional approach that is linear and fixed. This is what we used for our final group project. On Github, you are allowed to make tasks called “issues” and assign them to someone. This is how we completed tasks and progressed in our work in an efficient way as a team working remotely. I think that this style worked fairly well because our team liked to work at different times, and would run into problems often. Following issue driven project management, we would create a new issue for the problems that would come up and solve them as we went. This way, we would be able to always have a website that would be progressing. 


I think that this in particular worked well for code, but I can see myself using this project management style for another project that does not involve web applications, but follows a similar format. For example, I had another class that had a group project that I believe this method would have worked well for. For the most part, we would work on issues that we self assigned such as making slides, finding pictures for the presentation, gathering resources, etc. But once we encountered a problem, instead of staying stuck trying to solve it individually, it might have been a better idea to ask a group chat for help and make it a new “issue”. This brings attention to the most pertinent task to complete and makes sure that it gets done in a timely manner.
